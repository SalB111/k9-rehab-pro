VSI OpenVMS Calling Standard

Document Version: 4.0 Operating System: VSI OpenVMS Last Updated: June 2024

Preface

This document defines the requirements and conventions that govern the interaction between procedures executing on the OpenVMS operating system across VAX, Alpha, I64, and x86-64 architectures.

This standard establishes the calling mechanisms, argument passing conventions, register usage rules, and condition handling procedures that enable multilanguage interoperability across all supported OpenVMS platforms.

Note
This document is under engineering change order (ECO) control. Changes to this standard must be approved through the formal ECO process.

About VSI

VMS Software, Inc. (VSI) is the company that develops and supports the OpenVMS operating system. VSI continues to evolve OpenVMS to support modern hardware architectures while maintaining backward compatibility with existing applications.

Intended Audience

This document is primarily intended for developers of compilers and debuggers who need to understand the calling conventions used on OpenVMS systems. It is also relevant to systems programmers who write code at the procedure-call interface level.

Document Structure

This document is organized as follows:

  • Chapter 1 provides an introduction and key definitions
  • Chapters 2–5 describe architecture-specific calling conventions for VAX, Alpha, I64, and x86-64
  • Chapter 6 covers signature information for translated images
  • Chapters 7–8 define argument data types and descriptors
  • Chapter 9 describes the condition handling mechanism
  • Appendices A–C provide supplementary information

Chapter 1. Introduction

1.1 Applicability

The interfaces, methods, and conventions specified in this manual are primarily intended for use by implementers of compilers, debuggers, and other run-time tools for the OpenVMS operating system.

The conventions described here apply across all supported OpenVMS architectures: VAX, Alpha, I64 (Itanium), and x86-64. Architecture-specific details are provided in the relevant chapters.

1.3 Goals

The calling standard is designed to meet the following goals:

  • Enable interoperability between routines written in different programming languages
  • Support efficient procedure calls while maintaining adequate debugging support
  • Provide a consistent framework across different hardware architectures
  • Define clear conventions for argument passing, register usage, and stack management
  • Establish condition handling mechanisms for error recovery

1.4 Definitions

The following terms are used throughout this standard:

Procedure
A body of code that can be invoked via a call instruction and returns control to the caller upon completion. A procedure may accept arguments and return a function value.
Procedure Value
A value that uniquely identifies a procedure and can be used to call that procedure. On 64-bit systems, addresses are 64-bit values, though applications may restrict them to 32-bit representations for compatibility.
Bound Procedure
A procedure value that includes both the code address and an environment pointer, allowing the procedure to access its enclosing lexical scope.
Call Frame
The region of the stack allocated for a single procedure activation. The call frame contains saved registers, local variables, and temporary storage.
Condition Handler
A procedure that is invoked when an exception or signal occurs during execution. The handler may correct the error condition, unwind the stack, or resignal.
Stack Unwinding
The process of successively removing call frames from the procedure call stack, invoking cleanup handlers as each frame is removed.

Chapter 2. OpenVMS VAX Conventions

2.1 Register Usage

The VAX architecture provides 16 general-purpose registers (R0–R15) and a Processor Status Longword (PSL). The calling standard partitions these registers into specific categories based on their usage across procedure calls.

2.1.1 Scalar Processor Registers

The following table summarizes the register usage conventions for the VAX scalar processor:

Register Usage Preserved
R0 Function value return (low-order) No
R1 Function value return (high-order) No
R2–R11 General purpose, callee-saved as specified in procedure entry mask Per mask
R12 (AP) Argument Pointer Yes
R13 (FP) Frame Pointer Yes
R14 (SP) Stack Pointer Yes
R15 (PC) Program Counter N/A

2.2 Stack Usage

The VAX call stack grows from high addresses toward low addresses. The stack pointer (SP) always points to the top (lowest address) of the stack. The calling standard requires that SP be longword aligned (divisible by 4) at all times.

Important
Failure to maintain proper stack alignment may cause unpredictable behavior, including memory access violations and incorrect function return values.

2.3 Calling Sequence

The VAX architecture provides the CALLG and CALLS instructions for procedure invocation: 

CALLS   #n, procedure    ; Call with arguments on stack
CALLG   arglist, procedure ; Call with argument list in memory

Both instructions create a call frame on the stack, save registers specified in the procedure's entry mask, and transfer control to the target procedure. The RET instruction returns control to the caller by restoring the saved registers and removing the call frame.

Chapter 5. OpenVMS x86-64 Conventions

5.1 Register Usage

The x86-64 architecture provides 16 general-purpose registers, 16 SSE registers, and various system registers. The OpenVMS calling standard defines specific conventions for register preservation across procedure calls.

5.1.1 General-Purpose Register Classes

Registers are classified by their behavior across procedure calls:

  • Scratch registers — may be modified by the called procedure and are not preserved
  • Preserved registers — must be saved and restored by any procedure that modifies them
  • Special registers — have dedicated functions defined by the architecture or calling standard
Register Usage Class
RAX Return value, scratch Scratch
RBX General purpose Preserved
RCX 4th integer argument, scratch Scratch
RDX 3rd integer argument, scratch Scratch
RSP Stack pointer Special
RBP Frame pointer (optional) Preserved
RSI 2nd integer argument, scratch Scratch
RDI 1st integer argument, scratch Scratch
R8–R11 5th–8th arguments, scratch Scratch
R12–R15 General purpose Preserved

5.6 Procedure Call and Return

The x86-64 CALL instruction pushes the return address onto the stack and transfers control to the target procedure. The RET instruction pops the return address and transfers control back to the caller. 

; Example calling sequence
    sub     rsp, 32          ; Allocate shadow space
    mov     rcx, param1      ; 1st parameter in RCX
    mov     rdx, param2      ; 2nd parameter in RDX
    mov     r8,  param3      ; 3rd parameter in R8
    mov     r9,  param4      ; 4th parameter in R9
    call    target_procedure
    add     rsp, 32          ; Deallocate shadow space
Note
The OpenVMS x86-64 calling convention requires 16-byte stack alignment at the point of the CALL instruction. The 32-byte shadow space (home area) must always be allocated by the caller, even for procedures with fewer than 4 parameters.

Chapter 9. OpenVMS Conditions

9.1 Condition Values

OpenVMS condition values are 32-bit longword values that encode information about the facility that generated the condition, the specific condition, and its severity. The format is:

Bits Field Description
31:28 Control Control flags
27:16 Facility Facility number identifying the subsystem
15:3 Message Message number within the facility
2:0 Severity Severity level (see below)

Severity codes:

Code Value Meaning
0WarningOperation completed with qualification
1SuccessOperation completed successfully
2ErrorOperation not completed
3InformationInformational message only
4FatalSevere error, operation not recoverable

9.2 Condition Handlers

A condition handler is a procedure that is invoked when an exception or software-signaled condition occurs. Each call frame may optionally designate a condition handler. When a condition is signaled, the system searches the call stack for an established handler, starting with the frame in which the condition was raised.

A condition handler receives two arguments:

  1. Signal array — contains the condition value, signal-specific parameters, and the PC/PSL at the point of the signal
  2. Mechanism array — contains information about the state of the process at the time of the signal, including register contents

The handler can take one of the following actions:

  • Continue — correct the error and resume execution at the point of the signal
  • Resignal — decline to handle the condition; the system continues searching for another handler
  • Unwind — terminate one or more procedure activations and transfer control to a previous frame
